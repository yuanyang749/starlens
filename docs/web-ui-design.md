# Starlens Web UI 设计文档

> **状态：设计参考。** 本文记录视觉与交互原则，当前页面结构以实现和站内用户文档为准。

## 1. 设计目标

Web 端应当是一个长期可用的个人工具，而不是营销页或花哨的 AI 演示站。

核心目标：

- 快速找回仓库
- 低噪音、高可读
- 中高信息密度
- 支持长时间使用不疲劳
- 兼顾搜索、整理、AI 辅助和配置管理

整体气质定义为：

**安静专业、克制现代、偏工具化。**

## 2. 设计原则

### 2.1 功能优先

- 第一屏直接进入可用状态
- 不使用 hero、横幅、插画、营销式口号区
- 页面重点是搜索、结果、详情和整理动作

### 2.2 中高密度但不压迫

- 一屏要能看到足够多的结果
- 留白要服务于可读性，而不是追求“空”
- 通过版式层级控制信息负担，而不是靠大面积空白

### 2.3 低噪音视觉

- 中性色为主
- 单一强调色用于选中、主操作、焦点和状态提示
- 避免大面积彩色背景和多强调色竞争

### 2.4 稳定布局

- 主要区域在桌面端保持稳定，不频繁跳转页面
- 搜索、结果和详情尽量放在同一工作台中完成

## 3. 风格方向

### 3.1 视觉气质

- 安静专业
- 不花哨
- 不故作 AI 感
- 不做重营销风格
- 有一定高级感，但不抢功能的戏

### 3.2 颜色策略

推荐方向：

- 基础色：白、浅灰、深灰、近黑
- 分层色：偏冷中性灰
- 强调色：单一冷色，建议偏蓝灰或偏青灰

避免：

- 紫色主导
- 高饱和渐变
- 彩虹状态色
- 大面积深色背景压迫感

### 3.3 字体与图标

- 字体应选择现代、中性、屏幕可读性高的无衬线字体
- 代码、token、时间戳等信息可用等宽字体
- 图标以线性风格为主，笔画统一
- 图标是辅助识别，不应成为主视觉噪音

### 3.4 组件风格

- 小圆角：建议 `6-8px`
- 极轻阴影：更多依赖边框和底色层次
- 边框应克制、细且稳定
- 动效只用于反馈，不用于装饰

## 4. 信息架构

推荐采用**混合布局**：

- 顶部栏：全局能力
- 左侧栏：导航与常用过滤入口
- 中间主区：结果列表
- 右侧面板：详情与编辑

### 4.1 顶部栏

包含：

- 全局搜索输入框
- 同步按钮
- AI 入口
- 快捷过滤切换
- 用户头像 / 设置入口

设计要求：

- 高度稳定，不宜过高
- 搜索框是视觉中心
- 顶栏不做复杂渐变和装饰背景

### 4.2 左侧栏

用于承载长期稳定的导航和预设筛选：

- All Stars
- Favorites
- Recent
- By Language
- By Tag
- AI Search
- Tokens
- Settings

设计要求：

- 导航项简洁、可扫读
- 当前选中状态明确
- 可折叠，但默认展开

### 4.3 中间主区

结果区采用**列表优先**，不使用大卡片流。

每条结果建议展示：

- 仓库名
- owner
- 一句话摘要
- star 数
- language
- 更新时间
- tag
- 是否收藏
- 是否有备注

设计要求：

- 每条结果高度适中
- 支持 hover、选中、键盘上下切换
- 列表项不应像博客卡片或内容 feed

### 4.4 右侧详情面板

选中仓库后展示：

- 仓库全名
- GitHub 链接
- 摘要
- 元信息
- 标签管理
- 备注编辑
- AI 总结
- 相关操作

设计要求：

- 不单独跳转新页面作为主路径
- 面板应支持滚动，但标题与主要操作固定
- 信息分组清晰，避免一整块长文案堆叠

## 5. 关键页面设计

### 5.1 主工作台

这是首页，也是主界面。

页面结构：

- 顶部：搜索与全局动作
- 左侧：导航与固定过滤
- 中间：搜索结果列表
- 右侧：仓库详情

首页不需要欢迎语和空洞介绍，直接进入工具状态。

### 5.2 AI 搜索面板

AI 搜索应当是对主搜索的增强，而不是另起一个花哨聊天页。

建议形式：

- 可以在顶部搜索框旁切换为“AI 模式”
- 或在右侧开一个 AI 辅助面板

展示内容：

- 用户问题
- AI 推荐的候选仓库
- 每个候选的推荐理由
- 一段简短总结

设计要求：

- 不模仿通用聊天产品
- 输出要结构化，而不是一长段散文
- 保持和主结果列表的联动

### 5.3 AI Provider 设置页

这是偏工程化的配置页，风格应更冷静、明确。

页面内容：

- Provider 列表
- 新增 provider
- 编辑 provider
- 默认 provider 切换
- 校验状态
- 模型列表读取

每张配置卡或表单应明确显示：

- 协议类型
- 显示名称
- 模型
- 是否启用
- 是否默认
- 最近一次校验结果

### 5.4 Token 管理页

用于管理 CLI / agent token。

应突出：

- token 名称
- 创建时间
- 最近使用时间
- 权限说明
- 启用 / 撤销操作

设计要求：

- 安全感强
- 避免把 token 文本长时间暴露在页面
- 撤销操作要明确且可确认

## 6. 搜索结果的呈现规则

结果列表的信息优先级固定如下：

1. 仓库名
2. owner / 作者
3. 仓库用途摘要
4. star 数、language、更新时间
5. 用户标签与备注状态

要求：

- 仓库名最醒目，但不要过大
- 摘要最多显示两行
- 元信息尽量一行解决
- 标签用低噪音样式，不应过于彩色

## 7. 交互细节

### 7.1 搜索

- 输入后快速返回结果
- 支持键盘回车、上下切换
- 支持保留最近一次筛选条件

### 7.2 过滤

- 常用过滤器常驻可见
- 次级过滤器可折叠
- 清空过滤条件要简单直接

### 7.3 编辑

- 标签、备注、收藏尽量原地完成
- 保存反馈轻量明确
- 不频繁弹窗打断流程

### 7.4 同步

- 同步状态可见
- 正在同步时要有轻量进度反馈
- 同步失败时要有清晰错误信息

## 8. 响应式策略

### 桌面端

主工作模式，采用三栏结构：

- 左侧导航
- 中间结果
- 右侧详情

### 平板端

- 左侧栏可折叠
- 右侧详情可切为抽屉或覆盖层

### 手机端

- 顶部搜索优先
- 左侧导航改为抽屉
- 结果列表单列
- 详情改为独立层或底部抽屉

移动端不追求桌面完整体验，但核心搜索与查看详情必须顺畅。

## 9. 不应该做的事

- 不做营销首页风格
- 不做大面积渐变或炫光背景
- 不做紫色主导的 AI 风格界面
- 不做大卡片瀑布流
- 不做像聊天产品那样把 AI 作为唯一主界面
- 不做过度动画
- 不做一堆彩色标签造成视觉噪音

## 10. 设计结论

Web 端应被设计成一个：

**安静、可信、效率优先、可长期使用的个人 GitHub 收藏检索工作台。**

如果只能用一句话描述最终观感，应当是：

**不像营销页，不像玩具，不像花哨 AI 产品，而像一个成熟、克制、很好用的个人知识工具。**
