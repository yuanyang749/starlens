# App 工作台页面重构设计方案

## 1. 背景

当前 Starlens app 工作台页面已经具备基础的仓库搜索、筛选、同步、详情、标签与备注能力，但页面骨架仍是偏卡片化的双栏结构，与设计稿 [`design-assets/starlens-workbench-concept-21x9.png`](/Users/wangyuanyang/Documents/Myproject/Playground/starlens/design-assets/starlens-workbench-concept-21x9.png) 的桌面工作台形态存在明显偏差：

1. 设计稿是高密度、桌面优先的三段式工作台：顶部工具栏 + 左导航 + 中央视图区 + 右详情面板。
2. 当前实现是 marketing 风格更强的卡片式双栏页面，信息密度、边界形态、交互层级都不匹配。
3. 当前核心页面文件 [`apps/web/src/components/workbench-view.tsx`](/Users/wangyuanyang/Documents/Myproject/Playground/starlens/apps/web/src/components/workbench-view.tsx) 已承载过多状态、请求、布局与交互职责，超出项目对单文件规模与清晰边界的要求。

本次任务的目标不是新增业务功能，而是在不改变现有业务流程的前提下，把工作台页面重构为严格贴近设计稿的桌面工作区。

## 2. 用户确认的边界

### 2.1 还原精度

用户已明确选择：

1. 桌面端按设计稿高精度复刻。
2. 优先服从设计稿的布局骨架、留白比例、信息分区与控件位置。
3. 允许为了代码可维护性做实现层拆分，但不允许把这次工作扩展成产品逻辑重写。

### 2.2 响应式范围

用户已明确选择：

1. `桌面优先`。
2. 桌面端尽量贴近设计稿。
3. 移动端只做合理降级，保证可用、不崩、不出现明显横向滚动，不追求视觉等比复刻。

### 2.3 业务边界

本次重构只改布局、视觉表达与组件边界，不改以下行为的业务流程：

1. 搜索、筛选和排序。
2. URL 参数同步。
3. 仓库列表加载与详情加载。
4. GitHub 同步。
5. 收藏切换。
6. 标签新增与删除。
7. 备注自动保存。

本次也不新增以下功能：

1. AI Search 新流程。
2. 批量操作。
3. 高级筛选弹层逻辑扩展。
4. 详情区新业务动作。

## 3. 现状问题

### 3.1 结构与设计稿不一致

当前页面结构主要问题如下：

1. 顶部区域为两排头部信息，设计稿则是单排高密度工具栏。
2. 左侧侧栏更接近后台路由菜单，设计稿则是带分组和状态卡的工作台导航。
3. 主列表是纵向仓库卡片，设计稿是表格型资源浏览区。
4. 右侧详情区是多张信息卡片堆叠，设计稿则是连续性的检查面板。

### 3.2 代码架构坏味道

当前实现存在以下坏味道：

1. 晦涩性：单文件混合纯函数、请求逻辑、URL 同步、乐观更新、布局渲染与细节 UI。
2. 脆弱性：视觉改动容易牵扯状态与请求行为，修改成本高。
3. 不必要的复杂性：简单的视觉模块没有清晰拆分，导致后续演进变得困难。

这些问题直接影响这次高精度还原设计稿的稳定性，因此重构必须包含适度的结构拆分。

## 4. 设计目标

本次重构完成后，页面应满足以下目标：

1. 桌面端视觉骨架与设计稿一致。
2. 顶部工具栏、左导航、中央视图区、右侧详情面板四大区域边界清晰。
3. 页面主文件明显瘦身，视觉模块按职责拆分。
4. 所有现有核心交互可继续工作。
5. 移动端能够合理折叠为单列或双层结构，但不作为本次主要视觉验收对象。

## 5. 信息架构设计

### 5.1 顶部工具栏

顶部工具栏负责承载全局与高频动作：

1. 左侧品牌标识与名称。
2. 中部搜索输入。
3. 右侧操作区：`Sync now`、`AI Search`、`Filters`、设置按钮、用户头像入口。

视觉原则：

1. 单排布局。
2. 边框与阴影克制。
3. 搜索栏作为视觉中心。
4. 主操作按钮使用蓝色强调态。

### 5.2 左侧导航栏

左侧导航不再只是当前路由映射，而是改造成工作台结构导航：

1. `WORKBENCH`
   - `All Stars`
   - `Favorites`
   - `Recent`
2. `DISCOVER`
   - `Languages`
   - `Tags`
   - `AI Search`
3. `TOOLS`
   - `Tokens`
4. `SYSTEM`
   - `Settings`

底部固定 `Last sync` 状态卡，展示最近同步时间、总体数量与状态反馈。

### 5.3 中央视图区

中央主区采用表格型资源浏览界面：

1. 顶部二级过滤条：勾选框、局部搜索输入、列级筛选入口。
2. 表头：`Repository / Stars / Language / Updated / Tags / Favorite / Pin`
3. 表体：每行展示仓库名称、摘要、语言、标签与操作状态。
4. 底部分页区。

设计重点：

1. 选中行高亮。
2. 行与列的对齐感强于卡片层次感。
3. 标签胶囊、收藏图标、时间字段与设计稿保持接近的信息密度。

### 5.4 右侧详情面板

右侧详情区改造成固定面板式结构，内容顺序如下：

1. 仓库名称与关闭位。
2. GitHub 链接行。
3. 描述。
4. 指标胶囊：stars、language、license、updated、created、pushed、size、default branch。
5. Topics。
6. `My note`。
7. `AI summary`。
8. `Actions`。

视觉原则：

1. 模块连续，不做过度碎片化卡片堆叠。
2. 重点信息优先上移。
3. 辅助信息用低对比浅边框胶囊承载。

## 6. 组件拆分方案

为了同时满足高精度还原与可维护性，本次采用“容器 + 分区组件拆分”方案。

### 6.1 保留的容器层职责

页面容器负责：

1. URL 参数与筛选状态同步。
2. 仓库列表请求。
3. 选中仓库详情请求。
4. GitHub 同步请求。
5. 收藏、标签、备注等乐观更新逻辑。
6. 向子组件分发状态与回调。

### 6.2 新的视觉模块边界

建议拆分如下：

1. `workbench-view.tsx`
   - 页面容器。
   - 聚合筛选状态、请求状态、选中状态。
2. `workbench-topbar.tsx`
   - 顶部工具栏。
3. `workbench-sidebar.tsx`
   - 左侧导航与底部同步状态卡。
4. `repo-table-pane.tsx`
   - 中央视图区整体。
5. `repo-table-row.tsx`
   - 单行仓库渲染。
6. `repo-detail-panel.tsx`
   - 右侧详情面板。
7. `repo-detail-metadata.tsx`
   - 详情区指标胶囊网格。
8. `use-workbench-query-state.ts`
   - URL 参数与筛选状态逻辑。
9. `workbench-formatters.ts`
   - 日期、数字、URL 等纯函数。

如果实现中发现个别模块仍过大，可继续细拆，但不允许回退为单文件堆砌。

## 7. 样式与视觉系统策略

### 7.1 视觉语言

以设计稿为准，整体风格采用：

1. 冷白背景。
2. 浅灰边线。
3. 蓝色主操作按钮。
4. 低阴影、低毛玻璃感，强调清爽的桌面工作台气质。

不沿用当前页面过强的玻璃卡片感与营销式留白。

### 7.2 字体与信息密度

1. 保留现有技术栈下稳定可用的字体体系。
2. 强化表格场景的行内对齐、字段节奏与层级对比。
3. 顶部工具栏和表格采用更紧凑的 vertical rhythm。

### 7.3 交互反馈

只保留轻量微交互：

1. hover 背景或边框变化。
2. focus ring。
3. selected row 高亮。
4. 按钮按下与禁用态。

不引入重动画或复杂过场。

## 8. 响应式降级策略

本次移动端只做合理降级：

1. 左侧导航在中小屏下折叠。
2. 顶部工具栏允许换行或局部收纳。
3. 主表格在较窄宽度下改为卡片式或纵向列表式降级视图。
4. 右侧详情面板在移动端改为下方详情区或抽屉式呈现。

验收重点仍然是桌面端。

## 9. 风险与控制策略

### 9.1 主要风险

1. 组件拆分时破坏现有状态联动。
2. 表格化改造后，选中态、空态、分页、详情联动出现回归。
3. 右侧详情面板模块重排时遗漏现有字段。

### 9.2 控制办法

1. 先拆结构边界，再替换视觉布局。
2. 容器层继续统一掌握数据状态，子组件只做展示和事件回调。
3. 保留现有纯函数能力，并抽离到工具文件中复用。
4. 重构结束后通过构建、测试与手工关键路径验证共同兜底。

## 10. 验证方案

实现完成后至少执行以下验证：

1. 构建检查：确保 Next.js 构建通过。
2. 现有测试：运行 web 侧相关测试，确保没有明显行为回归。
3. 手工桌面验收：
   - 顶部工具栏布局与设计稿接近。
   - 左导航分组、状态卡、选中态正确。
   - 中央视图区具有表格化信息节奏。
   - 右详情面板信息顺序与视觉结构正确。
4. 手工交互验收：
   - 搜索。
   - 筛选。
   - 排序。
   - 选中仓库。
   - 收藏切换。
   - 标签增删。
   - 备注保存。
   - 同步反馈。
5. 移动端兜底验收：
   - 无明显布局崩溃。
   - 无严重横向滚动。
   - 关键功能入口仍可访问。

## 11. 结论

本次重构采用“桌面端高精度复刻设计稿 + 移动端合理降级 + 容器与视觉模块拆分”的方案。该方案在不改变现有业务流程的前提下，同时解决了页面结构与设计稿不一致、核心文件职责过重、后续维护成本偏高的问题，是当前任务范围内风险最低且质量最高的实施路径。
